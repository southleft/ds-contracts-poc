# Determinism: no AI in the conversion

**This is the most important document in the package.** If you internalize one
thing, internalize this.

## The principle

> The conversion between contract, code, and canvas is a **pure, deterministic,
> byte-reproducible function**. AI may author contracts and build the tooling.
> AI is **NEVER** part of the conversion itself.

## Why this is non-negotiable

The entire value proposition — "the contract is the single source of truth" —
depends on the conversion being deterministic. If an AI agent "renders" the
contract to a canvas by interpretation, then:

- The output varies run to run (probabilistic, not a function).
- Two surfaces (code and canvas) generated by two different AI passes can
  silently disagree — there is no single source of truth anymore, just two
  correlated guesses.
- You cannot *gate* it. You cannot say "this canvas is provably what the contract
  specifies," because an AI didn't compute it, it approximated it.

Determinism is what makes the guarantee real. Remove it and the project is just
another AI design tool.

## The evidence (this is proven, not asserted)

Gate: **`deterministic-roundtrip`** (`scripts/deterministic-roundtrip.mjs`, run
in the eval suite). It:

1. Runs `contract → canvas` through the real engine **twice** and asserts the
   built node trees are **byte-identical**. An AI in the loop could not make that
   guarantee; a pure function does, every time.
2. Runs `canvas → contract` (dump + propose) and recovers the anatomy — the loop
   closes.
3. Runs `contract → code` (emit React) from the same contract.

Output (verbatim, abbreviated):

```
1. contract → canvas  (the plugin engine, run twice)
   ✔ built CompositeModal both times; node trees byte-identical — DETERMINISTIC
2. canvas → contract  (dump the drawn set, propose a contract)
   ✔ recovered a contract from the drawn canvas — dialog+backdrop, composed
     ds.card instance, repeated ds.badge collection
3. contract → code  (emit React from the contract)
   ✔ emitted React from the same contract
✔ THE FULL LOOP RAN WITH ZERO AI — pure deterministic functions, byte-reproducible.
```

## The clincher (a true story from this project)

During the depth build, an AI (the assistant) *hand-built* a composite Modal on
the canvas through a raw Figma-API call, as a shortcut. It rendered an avatar as
an **oval** instead of a circle, and the AI had to manually patch the node.

The contract already said the avatar's `border-radius` is `radius.avatar →
radius.pill → 999` on a square avatar — a perfect circle *by construction*. The
deterministic emitter, driven by the contract, would never produce the oval. The
moment the AI stepped *outside* the deterministic pipeline to hand-build and
hand-patch, it introduced drift and had to fix it by hand — **which is exactly
the failure mode contracts exist to eliminate.**

The AI-built render was a *representation* that bypassed the source of truth.
That is the anti-pattern. Do not repeat it.

## The consequence for tooling (important architectural fact)

Running the deterministic emitter requires a runtime *inside* Figma. The engine
that emits + builds is ~288KB of generated script per full build. **No AI agent
tool can carry or run that** — not `figma_execute` (the Figma-console MCP), not
`use_figma` (the claude.ai Figma MCP). Both are the same execution model
(Plugin-API JavaScript, ~50KB cap, stateless per call), and both hit the payload
wall. This is not a limitation to route around with AI. It is the *reason the
plugin exists*: the engine lives in Figma, is fed a ~10KB contract, and runs the
conversion deterministically with no AI.

So the deterministic vehicles are:

- **The CLI** (`@ds-contracts/cli`) for `code ↔ contract` — deterministic, no AI.
- **The plugin** (DS Contracts Sync Runner) for `contract ↔ canvas` — the engine
  is baked in; you feed it a contract; it emits + builds deterministically, no
  AI, no network needed for the core path.

## What AI *is* allowed to do

- **Author or edit contracts** (a human or AI can write a `*.contract.json`; the
  schema and emitters then deterministically enforce and render it).
- **Build and improve the tooling** (the emitters, extractors, engine, plugin,
  tests) — this is where AI adds value.
- **Investigate and verify** (read the canvas, run gates, diagnose bugs).

AI must **never** be the thing that turns a contract into pixels, or pixels into
a contract. That is always a pure function.

If a future direction tempts you toward "just have the agent render it via
`use_figma` — it's high fidelity" — that temptation was explicitly raised and
explicitly rejected in this project (see `10-history.md`, the determinism
correction). Re-read this document before proposing it again.
